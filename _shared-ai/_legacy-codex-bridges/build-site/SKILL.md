# build-site — Webサイト一気通貫構築スキル

10ステップのWebサイト構築プロセスを **3フェーズ** に圧縮し、
対話的に高品質なサイトを生成する。

## コンセプト

従来のAIサイト構築は「1プロンプト1タスク」で10回以上のやり取りが必要。
このスキルは以下の原則で圧縮する:

- **計画とコピーは同時に考える**（分けると一貫性が崩れる）
- **SEOは後付けではなく最初から組み込む**
- **コード生成時にレスポンシブ・アクセシビリティも含める**
- **改善は `/web-audit` と連携して自動化**

---

## 実行フロー

### 入力の確認

`$ARGUMENTS` または対話から以下を収集する。不足分は質問して埋める:

| 項目 | 例 | 必須 |
|---|---|---|
| 業種 | パーソナルジム、税理士事務所、EC | ✅ |
| ターゲット | 30代女性、中小企業経営者 | ✅ |
| 主な目標 | リード獲得、予約、売上、ポートフォリオ | ✅ |
| トーン | プロフェッショナル、フレンドリー、モダン、ラグジュアリー | 任意（デフォルト: モダン） |
| ページ数 | 1ページLP〜5ページサイト | 任意（デフォルト: 3ページ） |
| 出力先ディレクトリ | `client-projects/xxx/` or `reusable/xxx/` | ✅ |
| 参考URL・競合 | 任意 | 任意 |
| ブランドカラー | 任意（なければ提案） | 任意 |

## Phase 1: 戦略設計（Plan）

計画モードで以下を一括出力し、ユーザー承認を得る。

### 1.1 サイト戦略
- ページ構成（どのページが必要か）
- 各ページの目的とゴール
- サイトマップ（シンプルな階層構造）
- ユーザーフロー（訪問→コンバージョンの導線）

### 1.2 コンテンツ設計
各ページについて:
- セクション構成（ヒーロー、信頼、サービス、CTA等）
- 各セクションのコピー（見出し・本文・CTA文言）
- SEOキーワード（タイトル・メタディスクリプション・H1-H3）

### 1.3 ビジュアル方針
- カラーパレット（プライマリ・セカンダリ・アクセント・背景・テキスト）
- フォントペアリング（見出し・本文）
- ボタンスタイル・角丸・シャドウ
- スペーシング・レイアウトの基本方針
- 全体の雰囲気（モードボード的な言語化）

### 出力フォーマット

```markdown
# サイト戦略: [業種名]

## ページ構成
1. ホーム — [目的]
2. サービス — [目的]
3. About — [目的]
...

## ユーザーフロー
[訪問] → [ヒーローで価値提示] → [サービス確認] → [信頼獲得] → [CTA]

## コンテンツ設計

### ホームページ
#### ヒーローセクション
- H1: 「...」
- サブ見出し: 「...」
- CTA: 「...」
...

## ビジュアル方針
- カラー: #xxx (プライマリ), #xxx (セカンダリ), ...
- フォント: [見出し] / [本文]
- トーン: [雰囲気の説明]
```

**ユーザーに確認**: 「この戦略で進めますか？修正したい部分があれば教えてください。」

## Phase 2: コード生成（Build）

Phase 1の承認後、以下を一気に生成する。

### 2.1 ベースファイル生成

出力先ディレクトリに以下を作成:

```
[output-dir]/
├── index.html          # ホームページ
├── services.html       # サービスページ（必要な場合）
├── about.html          # Aboutページ（必要な場合）
├── css/
│   └── style.css       # 全ページ共通スタイル
├── js/
│   └── main.js         # インタラクション（スクロール、ハンバーガー等）
└── images/             # プレースホルダー指示
```

### 2.2 コード品質基準

生成するコードは以下を満たすこと:

**HTML**
- セマンティックタグ（header, nav, main, section, footer）
- meta description, OGP, viewport設定
- 構造化データ（Schema.org LocalBusiness等、業種に応じて）
- alt属性（プレースホルダーでも意味のあるテキスト）

**CSS**
- CSS変数でカラー・フォント・スペーシングを管理
- モバイルファースト（min-widthメディアクエリ）
- ブレークポイント: 768px（タブレット）、1024px（デスクトップ）
- `:root` で全テーマ変数を定義

**JavaScript**
- バニラJS（フレームワーク不要の場合）
- スムーズスクロール
- モバイルメニュー
- スクロールアニメーション（IntersectionObserver）
- 最小限・軽量

**SEO（最初から組み込み）**
- 各ページのtitle, meta description
- H1は1つ、H2-H3で階層化
- 画像alt属性
- 内部リンク構造
- canonical URL

**アクセシビリティ**
- aria-label（ナビゲーション、ボタン）
- フォーカス可視性
- カラーコントラスト比4.5:1以上
- キーボードナビゲーション対応

### 2.3 ページ間の一貫性

- 共通ヘッダー・フッター（全ページ同一構造）
- ナビゲーションの現在ページハイライト
- CTAの統一（文言・色・配置）
- フォントとカラーの統一（CSS変数で管理）

## Phase 3: 品質チェック & 改善（Review）

コード生成後、自動的に以下を実行:

### 3.1 セルフレビュー

生成したコードを以下の観点でチェック:

| チェック項目 | 基準 |
|---|---|
| レスポンシブ | 320px〜1440pxで崩れないか |
| SEO | title, meta, H1, alt全て埋まっているか |
| アクセシビリティ | aria, コントラスト, キーボード対応 |
| パフォーマンス | 不要なCSS/JSがないか、画像最適化の指示があるか |
| CTA | 各ページに明確なCTAがあるか |
| コード品質 | クリーンなクラス名、DRY、コメント適切か |

### 3.2 web-audit連携（任意）

ユーザーに提案: 「`/web-audit` で詳細なスコアリングを実行しますか？」

### 3.3 改善提案

レビュー結果をもとに:
- 自動修正可能なものはその場で修正
- 画像やコンテンツなどユーザー判断が必要なものはリスト化

## 既存テンプレートの活用

`reusable/` ディレクトリに既存テンプレートがある場合:

1. `reusable/Dark Luxury/` 等の既存LPを確認
2. 業種・トーンに近いテンプレートがあれば、ベースとして活用を提案
3. ゼロから作るより差分適用の方が品質・速度ともに上

**ユーザーに確認**: 「既存テンプレート [名前] をベースにしますか？それとも新規で作りますか？」

## 注意事項

- 画像はプレースホルダー指示のみ（実画像はユーザーが用意）
- ドメイン・ホスティングの設定はスコープ外
- フォーム送信先は仮のaction属性（実装はユーザーまたは別タスク）
- 外部サービス（Google Analytics等）の設置コードは指示のみ
- `.env` にはシークレットを含めない
